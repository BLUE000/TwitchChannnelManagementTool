# TwitchChannelManagementTool 結合テスト仕様書 (ドラフト)

本ドキュメントは、基本設計に対応する「結合テスト（Integration Test）」の自動化および検証項目を定義するものです。外部システムとの通信をスタブ・モックに差し替え、ローカル環境（インターネット接続・外部連携アプリ起動不要）で自動実行可能な検証を行います。

---

## 1. 結合テスト方針
* **対象範囲**: 各モジュール間（UI ➔ コア ➔ 各モジュール ➔ プラグイン ➔ 外部中継器）の協調動作、シグナル/イベントフローの疎結合連携の検証。
* **テスト環境**: 
  - 本物のTwitch API、OBS、音声合成ソフトは使用せず、テストモード専用の「スタブ（Stubs / Mocks）」に差し替えます。
  - テストビルドターゲット（`run_integration_tests.exe`）を構築し、テストコードで各中継器の正常性を自動検証します。

---

## 2. 外部サービススタブ（テストモード用）

自動結合テストを実現するため、以下のスタブモジュールを実装し、テスト実行時に注入（Dependency Injection）します。

1. **`MockTwitchServer` (Twitch EventSub スタブ)**
   - インターネットを介さず、任意のタイミングで「チャット受信イベント」「チャンネルポイント引き換えイベント」をローカルでシミュレート・発火させ、本体コアへ配信する。
2. **`MockTtsReceiver` (TTS スタブ)**
   - 棒読みちゃん用TCPサーバーのダミーとして動作、またはVOICEVOXのREST HTTP応答のダミーとして動作する。
   - `TtsManager` から実際に送信された TCPパケットや HTTPリクエストを受信し、ヘッダー情報や送信テキスト、各種パラメータをインメモリに記録してテストコードに提供する。
3. **`MockObsClient` (OBS ブラウザソース スタブ)**
   - `ObsHttpWebSocketServer` にローカル接続するダミーの WebSocket クライアント。
   - サーバーからブロードキャストされた JSON を受信し、受信メッセージリストに格納して検証可能とする。

---

## 3. 結合テストケース一覧

### 3.1. 下り方向通信およびイベント伝播テスト
* **IT_DWN_001: Twitchコメント受信 ➔ 画面表示・プラグイン伝播の結合テスト**
  - **前提**: `MockTwitchServer` およびダミープラグインが有効化されている。
  - **手順**: 
    1. `MockTwitchServer` からダミーコメント「こんにちは」を EventSub 経由で発火させる。
  - **検証内容**:
    - `TwitchEventCollector` がコメントを検出し、C++構造体 `TwitchComment` に正しく変換すること。
    - `SignalDispatcher` を経由して、接続されているダミープラグインの `onCommentReceived` メソッドが呼び出されること。
    - プラグイン内のシグナルによって、`MainWindow`（UI層）のコメント一覧表示が正しく更新されること。

### 3.2. 上り方向通信および外部連携の結合テスト
* **IT_UP_001: プラグイン ➔ コア ➔ TTS連携（棒読み/VOICEVOX）の結合テスト**
  - **前提**: `MockTtsReceiver` がローカルで起動しており、TTS設定が有効。
  - **手順**:
    1. テストコードから、ダミープラグイン内のロジックを通じて、コアへのカスタムイベント `PluginNotifyEvent(TypeTtsSpeak)` を `QCoreApplication::postEvent()` 経由で発行する。
  - **検証内容**:
    - `AppController` の `onNotifyEvents()` がこのイベントを安全にキャッチすること。
    - コアから `TtsManager` に命令が伝わり、選択されているTTSエンジンの送信処理が走ること。
    - `MockTtsReceiver` に、指定された発話文字列およびパラメータ（速度、ピッチ、音量）が100%正確な値で到達していること。

* **IT_UP_002: プラグイン ➔ コア ➔ OBS WebSocketサーバーの結合テスト**
  - **前提**: `MockObsClient` が `ObsHttpWebSocketServer` にWebSocket接続を確立している。
  - **手順**:
    1. ダミープラグインから `PluginNotifyEvent(TypeObsBroadcast)` を発行する。
  - **検証内容**:
    - コアのイベントハンドラを経由して、WebSocketサーバーが受信クライアントにJSONデータを配信すること。
    - `MockObsClient` が受け取ったJSONデータが、プラグインで発行された描画・演出データ（座標、動画パス等）と完全に一致すること。
* **IT_UP_003: プラグイン ➔ コア ➔ Discord Webhook 送信の結合テスト**
  - **前提**: テスト用のローカルWebサーバー（Mock Webhookエンドポイント）が待機中であること。
  - **手順**:
    1. ダミープラグインから `ICoreContext::postDiscordWebhook()` を介して、Mock Webhook URL宛てにテスト用JSONデータを送信する。
  - **検証内容**:
    - `AppController` が非同期で HTTP POST リクエストを発行し、Mock エンドポイントにリクエストが正常到達すること。
    - 受信した JSON データが、プラグインから送信された内容と完全一致すること。

### 3.3. プラグイン動的ライフサイクルテスト
* **IT_LFC_001: プラグイン有効/無効の動的ロード切り替えテスト**
  - **手順**:
    1. `SettingsTab`（UI層）の設定変更によって、プラグインAを「無効化」し設定を保存する。
    2. プラグインAを再ロード、または別のプラグインBを「有効化」する。
  - **検証内容**:
    - 無効化されたプラグインAのウィジェットが `MainWindow` のサイドタブから即座に削除されること。
    - `SignalDispatcher` とプラグインAの間のQtシグナル接続が完全に解除され、以降のイベント送信でプラグインAが呼び出されないこと。
    - 有効化されたプラグインBのウィジェットがサイドタブに自動追加され、シグナルが正常にバインドされること。
* **IT_LFC_002: アンロード時の安全破棄シーエンステスト**
  - **検証内容**:
    - プラグイン無効化の際、`QPluginLoader::unload()` を呼ぶより前に、プラグイン提供の `QWidget` が確実に削除（デストラクタが動作）されていることを検証する。
* **IT_LFC_003: UI上の[上へ]/[下へ]ボタンによる並び替え結合テスト**
  - **手順**:
    1. プラグインAとプラグインBを有効化し、サイドバー表示順が「A ➔ B ➔ 設定」であることを確認する。
    2. 設定画面のリストで「B」を選択し、`[上へ]` を押して保存する。
  - **検証内容**:
    - アプリケーションが再起動せず、サイドバーの表示順が「B ➔ A ➔ 設定」に動的に入れ替わること。
    - 左サイドバーのクリック時の `QStackedWidget` の切り替えページインデックスが、並び順に連動して正常に再構成されていること。
* **IT_LFC_004: OBS HTTPサーバーによる静的ファイル配信テスト**
  - **手順**:
    1. `ObsHttpWebSocketServer` に対し、テストクライアントから `GET /overlay/[プラグイン名]/default/index.html` をリクエストする。
  - **検証内容**:
    - レスポンスコード `200 OK` が返ること。
    - レスポンスヘッダーの `Content-Type` が `text/html` となっており、返される内容がディスク上のファイルと完全一致すること。
    - 存在しないファイルをリクエストした場合、`404 Not Found` が返ること。
